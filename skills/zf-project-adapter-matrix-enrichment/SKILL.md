# ZaoFu Project Adapter Matrix Enrichment

Use this skill when a workflow contains delivery matrix drafts generated by
`zf flow intake`, especially:

- `source-inventory.json`
- `capability-matrix.json`
- `acceptance-matrix.json`
- `test-matrix.json`
- `task-map.json`
- `real-e2e-matrix.json`

## Boundary

- Put project semantics in matrix artifacts, prompts, and skills.
- Do not ask runtime/kernel code to know product names, module names, provider
  choices, UI parity details, or domain-specific commands.
- Runtime may enforce schema, path safety, evidence existence, command execution,
  and resume/audit invariants only.

## Enrichment Workflow

1. Read `metadata.enrichment_contract` from each matrix draft.
2. Replace portable placeholders with source-backed project facts.
3. Preserve stable ids where possible; add new rows for discovered gaps.
4. Link every blocking capability to acceptance, test, and task rows.
5. Write updated matrices as artifacts; do not mutate kernel truth directly.
6. If facts are missing, mark the row `blocked` or `needs_input` and include
   `reason` plus `evidence_refs`; do not mark it passed.
7. For a completed matrix, set top-level `status` to `ready` and
   `metadata.enrichment_contract.status` to `fulfilled`.

When the Plan Synth result protocol is active, return enriched matrices through
`plan_ports` instead of overwriting intake files:

```json
{
  "plan_ports": [
    {
      "logical_name": "acceptance_matrix",
      "schema_version": "acceptance-matrix.v1",
      "body": {
        "schema_version": "acceptance-matrix.v1",
        "status": "ready",
        "metadata": {
          "enrichment_contract": {"status": "fulfilled"}
        },
        "acceptance": []
      }
    }
  ]
}
```

The Kernel materializes each body as an immutable sidecar and only validates
schema/readiness/ref/digest/currentness. Content quality remains the Plan
agent's responsibility.

Complete enrichment before the Task Map/Task Contract snapshot is admitted.
Impl and Verify consume immutable current-package ports; they must not rewrite
portable matrix drafts after dispatch. For a light topology without a Planner
stage, the intake/adapter path must either provide enriched matrices before
`task_map.ready` or leave the required port unresolved so dispatch fails
closed. Do not transfer matrix-authoring authority to the implementation lane.

## Required Matrix Updates

- Source inventory: cite concrete files, directories, modules, endpoints,
  commands, screens, or specs.
- Capability matrix: map source behavior to target behavior and identify gap
  status.
- Acceptance matrix: state user-visible or operator-visible acceptance criteria.
- Test matrix: include deterministic commands or evidence references.
- Task map: assign every blocking capability to role, lane/affinity, allowed
  paths, and expected evidence.
- Real E2E matrix: for live surfaces such as CLI, web, LLM, gateway, provider,
  TUI, dashboard, API, or browser, declare a real command or attach passing
  evidence.

## Real E2E Command Rules

- The runner executes only commands declared in `real-e2e-matrix.json`.
- Use commands that can run from the project root.
- Prefer Docker Playwright for browser checks when required by the repo.
- Use real provider/env probes when the objective requires live LLM/gateway
  validation.
- Do not use mock-only commands for release/full-parity validation unless the
  objective explicitly scopes the run to mocks.
- Do not mark a required real E2E row passed without `command` output or
  `evidence_refs`.

## Kind Guidance

- issue: focus inventory on reproduction path, failing behavior, root-cause
  files, regression test, and fix task.
- prd: focus inventory on spec, product surfaces, user flows, API/data model,
  demo evidence, and release checks.
- refactor: focus inventory on source modules, target module mapping, parity
  gaps, compatibility tests, real integration surfaces, and final replacement
  readiness.

## Output Quality Bar

The enriched artifacts are good enough only when a downstream impl/verify/judge
agent can answer:

- what exact source behavior must be preserved or built;
- which task owns the work;
- which files/paths it may change;
- how to prove it is done;
- which real command or evidence proves high-risk surfaces.
