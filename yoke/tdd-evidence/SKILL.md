# TDD Evidence

先写失败的测试,再写让它通过的代码;修 bug 先用测试复现。测试是证明——
"看起来对了"不算完成。在 ZaoFu 里这不是风格偏好,而是**完成证据的生产线**:
你的 `dev.build.done` 会被 4 道机械审计消费(schema / attempt 绑定 / 代际
current / 契约 scope),verify reader 会复跑你的证据,复跑不过比没证据更重。

## 何时使用

- 实现任何新逻辑或行为;修任何 bug(Prove-It 模式);改任何既有功能;
  补任何边界处理。
- **不适用**:纯配置、文档、无行为影响的静态内容。

## 与 kernel 合约的配对

| 你产出的 | 谁消费 | 驳回条件 |
|---|---|---|
| 新测试 + 实现同 commit | verify reader 源码轴 | 只有实现没有测试 → gap finding |
| `dev.build.done` 的 evidence(测试命令+计数) | 完成审计 + verify 复跑 | 不可复跑 = 假证据,直接返工 |
| commit 前缀 `feat:`/`fix:` | 归档审计 | TDD 提交(测试+实现一体)用 `feat:`,不拆 `test:` |
| bug 修复的复现测试 | verify 回归轴 | 修复无对应回归测试 → rework |

## TDD 循环

```
   RED               GREEN                REFACTOR
 写失败测试  ──→  最小实现使其通过  ──→  绿灯保护下清理  ──→ (重复)
 测试必须失败      不过度设计            每步重跑测试
```

1. **RED**:先写测试,必须先失败——立即通过的测试什么也证明不了。
   留下那次失败输出,它是证据链第一环。**先读 task contract 的
   `verification` 字段**:验收命令是真相,你的测试对齐它,不发明
   自己的验收——contract 就是 ZaoFu 的"预先约定 seam":测试写在
   约定的公共边界上,不测内部结构。
2. **GREEN**:最小代码让它过。禁止顺手改 allowed_paths 之外的文件
   (scope gate 会拒);禁止为"以后可能用"预铺抽象。
3. **REFACTOR**:仅在全绿保护下做,且不扩大 changeset——5 行根因的
   修复不该动 10 个文件。

## Prove-It 模式(bug 修复)

收到 bug **不要先动手修**,先写复现测试:

```
bug 报告 → 写复现测试 → 测试失败(bug 确证)→ 实现修复
        → 测试通过(修复证明)→ 全量相关测试(无回归)
```

复现测试写在能触达 bug 的最低层 seam(单元 > 集成 > e2e);复现失败
输出与修复后通过输出都进完成汇报。

## 测试金字塔与尺寸决策

```
        /\      E2E (~5%)        完整用户流,真浏览器
       /--\     集成 (~15%)      跨边界(API/DB/文件系统)
      /----\    单元 (~80%)      纯逻辑,毫秒级
```

| 尺寸 | 约束 | 速度 |
|---|---|---|
| Small | 单进程、无 I/O、无网络、无 DB | 毫秒 |
| Medium | 可多进程、仅 localhost、无外部服务 | 秒 |
| Large | 可跨机、可外部服务 | 分钟 |

决策:纯逻辑无副作用 → 单元;跨边界 → 集成;关键用户流 → E2E
(只留关键路径)。**Beyonce 规则**:如果你在乎它,就给它上测试——
重构和迁移不负责接住你的 bug,你的测试负责。

## 好测试五律

1. **测状态,不测交互**:断言操作的*结果*,不断言内部调了哪些方法——
   交互断言在重构时碎一地,即使行为没变。
2. **测试里 DAMP 优于 DRY**:每个测试独立可读、自带完整故事;为了
   省几行输入形状把 setup 抽到共享 helper,读者就得跳着读。测试里的
   重复是可接受的。
3. **真实现 > fake > stub > mock**:测试用的真代码越多信心越高。
   只在真依赖太慢/不确定/有不可控副作用(外部 API/发信)时才 mock;
   全 mock 的测试绿着,产品塌着。
4. **Arrange-Act-Assert**:三段清晰分层;一个测试验证一个概念,
   多概念拆多测试。
5. **命名读起来像规格**:completing an already-completed task is a
   no-op 这样的名字,而不是 works / handles errors。

## 反模式表

| 反模式 | 问题 | 解法 |
|---|---|---|
| 测实现细节 | 重构即碎 | 测输入输出 |
| flaky(时序/顺序依赖) | 信任崩塌 | 确定性断言、状态隔离、pin 时间/种子 |
| 测框架行为 | 浪费 | 只测自己的代码 |
| **同义反复断言** | 期望值用与代码相同的方式重算(`expect(add(a,b)).toBe(a+b)`),构造性通过,永远不会不同意代码 | 期望值来自独立真相源:已知字面量、手工算例、规格 |
| snapshot 滥用 | 没人审的大快照 | 少用且每次变更过目 |
| 无隔离 | 单跑过合跑挂 | 各自 setup/teardown |
| 全 mock | 假绿 | 边界处才 mock |
| 借宿主环境 | CI=true 下必挂 | 不依赖 TTY/端口/全局缓存(r4 pnpm no-TTY 双踩) |

## ZaoFu 实战锚

- **自声明完成**是完成诚实文化第一违例:"实现完了"没有测试输出 =
  没完成。
- **矩阵空转**(F14):声称覆盖验收条款但点名不出对应测试——每条
  条款要能点名到测试名。
- **跨 lane 类型偏斜**:你改了共享类型(如必填字段)只测自己包,
  combined-tree typecheck 会在 candidate 集成时抓你;改共享契约本地
  先跑全仓 typecheck。
- 浏览器行为的运行时验证走 zf-browser-e2e-contract(Docker
  Playwright、bounded install、runner/product 失败分类),本技能不重复。

## 常见借口

| 借口 | 现实 |
|---|---|
| "先跑通再补测试" | 你不会补的;事后测试测的是实现不是行为 |
| "这太简单不用测" | 简单代码会变复杂;测试是行为文档 |
| "测试拖慢我" | 现在慢一点,之后每次改动都快 |
| "我手工测过了" | 手工测试不持久,明天的改动没人接住 |
| "只是个原型" | 原型会变成产品 |
| "再跑一遍确认下" | 代码没变,重跑同命令不增加任何信心 |

## Red Flags

- 写代码没有任何对应测试;测试首跑即过(可能没测到你以为的东西);
- "全部通过"但根本没跑;bug 修复无复现测试;
- 跳过/禁用测试让套件变绿;代码未变重复跑同一命令。

## Verification

- [ ] 每个新行为有对应测试,全部通过
- [ ] bug 修复含修复前失败的复现测试
- [ ] 测试名描述行为;无跳过/禁用
- [ ] 完成汇报含可复跑的测试命令与计数,且与 HEAD 绑定
- [ ] contract.verification 的命令亲手跑过一次
