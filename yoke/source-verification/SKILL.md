# Source Verification

对外部框架、API、协议写代码或做扫描结论之前,先验证来源。**训练记忆
不是来源**——它没有版本号,而框架每个大版本都在改 API。凭记忆写出的
"看起来对"的调用,是 verify 驳单和线上静默错误的高发源。

## 判定:什么时候必须查

- 调用**外部框架/库的 API**(参数、返回值、生命周期、默认值);
- 依赖**协议/格式细节**(HTTP 语义、JSONL、SDK 事件流);
- 做**版本相关**判断(某特性从哪个版本可用、废弃了什么);
- 扫描报告里写"X 系统支持/不支持 Y"这类事实宣称。

**不用查**:语言基础语法、仓内自有代码的行为(直接读代码/跑测试,
那是更高一级的权威)。

## 权威层级(高 → 低)

1. **目标仓现有代码与测试**——项目怎么用就怎么用;改约定要先上报;
2. **依赖清单钉住的版本 + 该版本的官方文档/CHANGELOG**;
3. **依赖源码**(文档缺失时直接读 node_modules / site-packages 实现);
4. 维护良好的社区资料(注明出处);
5. 训练记忆——**只配当假设,不配当依据**;用它起草可以,落盘前按
   1-3 验证。

## 流程

1. **定版本**:从依赖清单读(package.json / pyproject.toml / lockfile),
   不猜、不用"最新版"代替;
2. **先 grep 现网**:要引入的能力目标仓是否已有(符号/近似实现)——
   已有就复用或说明为何不用,别平行造轮子(借鉴类任务的高频翻车点);
3. **查权威来源**:按上面层级取证,记下出处;
4. **照来源实现**:用文档写明的模式,不混搭版本;
5. **冲突上报**:官方推荐与仓内惯例矛盾时,列出两边证据交决策
   (grill 的事实/决策分流),不静默择一。

## 宣称与出处(对接 evidence_refs)

- 非显然的外部用法,报告/完成事件里给出处:仓内证据用真实路径,
  外部证据用 `<scheme>:<value>` 语法(如 `url:https://...`、
  `doc:express@5-migration`),与 done-contract 的 ref 约定一致;
- 查不到权威来源又必须落盘的判断,显式标 **UNVERIFIED** 并说明
  依据什么假设——标注过的假设可以被 verify 定点复核,编造的"事实"
  只能靠事后炸。

## 反合理化

| 借口 | 现实 |
|---|---|
| "这个框架我熟" | 你熟的是训练截止时的版本。先对依赖清单里的版本号。 |
| "查文档太慢" | verify 驳一轮返工比查一次文档慢得多,而且烧真金。 |
| "大概率是这个 API" | 大概率 = evidence_refs 里没有出处。写 UNVERIFIED 或去查。 |
| "社区都这么写" | 社区写法常绑旧版本。有官方来源先用官方。 |
